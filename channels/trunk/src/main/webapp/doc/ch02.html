Channels basics

Channels is a collaborative Web application for building, analyzing and deploying Information Sharing Plans.

Each Information Sharing Plan has one development version (the only one that can be modified), possibly one production version (the only one participants can access to view their playbooks), and zero or more retired versions (prior production versions).

User roles

Channels recognizes three user roles: administrators, planners and Plan participants. Administrators assign planners and participants to plans hosted by a Channels instance. Planners jointly develop, review and deploy information sharing plans. Participants view their playbooks and respond to surveys from planners about issue resolutions.

An administrator can:

  • add new plans to Channels,

  • assign users to plans and change their user roles,

  • do what any planner or participant can do, but for all Plans.

A planner can:

  • view assigned plans (all versions),

  • modify the development version of an assigned Plan,

  • view the reports from an assigned Plan (all versions),

  • view all playbooks from an assigned Plan (all versions),

  • vote with other planners to put the development version of an assigned Plan into production,

  • launch, access and close online surveys of selected planners and participants about issues in an assigned Plan.

A participant can:

  • view his/her playbook from the production version of an assigned Plan,

  • access and respond to online surveys.

Logging in and selecting a Plan and segment

When accessing Channels, an unrecognized user is presented with a login screen.

After logging in, a user with planning privileges (a planner) accesses the Plan Editor. A participant is directed to his/her playbook (see section 3.15). For a planner, the editor opens on the development version of an assigned Plan.

When the Plan Editor is first accessed, Channels pre-selects a segment and a task within it. The task with its “receive” and “send” information flows are shown “collapsed” and summarized.

A planner can at any time select to see another version of the Plan, if any. A planner can also switch to another assigned Plan. Only the development version of a Plan can be modified; retired and production versions are “read-only” (all form fields are disabled, no survey can be launched).

A planner can at any time select which segment within a Plan to view or work on. The tasks and information flows of the selected segment are visualized as a flow map (a directed graph). The selected task is highlighted (by a gray rectangle) in the flow map.

Flow maps can be quite large. A planner can maximize a flow map to see more of it without needing to scroll. Whether maximized or not, a flow map can be zoomed out to fit within the available space.

Tasks and flows are drawn in red if they have known issues. Green and red traffic lights represent the start and end of the event phase covered by the segment. Tasks that are triggered by the start of the segment are connected to the green traffic light icon. Tasks that terminate the event phase are linked to the red traffic light.

Information flows (information shared), information needs (information needed for carrying out a task) and information capabilities (shareable information produced by carrying out a task) are shown as arrows. Single-headed arrows represent notifications. Double-headed arrows are requests, with the empty arrow head pointing to the task that consumes the information request, and with the full arrow head pointing to the task that consumes the answer (the requested information flows to it).

Dots indicate to-be-determined tasks. An arrow from a task to a dot represents an information capability of the task (information produced by the task that can be shared for use in other tasks). An arrow from a dot to a task represents an information need of the task (the task needs information possibly produced by other tasks).

Tasks with a blue background denote remote tasks, that is, tasks from another segment that are connected by sharing flows to tasks in the selected segment.

When a task is selected in a maximized flow map, the flow map is automatically de-maximized to show the summaries of the selected task and those of its flows.

Editing tasks and flows

When a task is expanded, its summary is replaced by a form with fields for the editable attributes of the task. The “sends” of the task (information shared by the task plus its information capabilities) and the “receives” of the task (information shared with the task plus its information needs) can also each be expanded into their edit forms.

A task is defined by first naming it. It is further defined by stating where it is to be executed, by whom, in what role, for what jurisdiction, and on behalf of what organization. Any of the location, agent, jurisdiction, role, and organization (entities) that specify a task can be left unnamed so as to imply that more than one agent can be assigned to the task. When the entities are named, the names can identify actual entities (Morritown, NJ, John Doe, ACME etc.) or types of entities (a city, a certified technician, a corporation etc.).

The execution of a task can be set to be “as a team” to indicate that all assignees are to work together when carrying out the task. By default, assignees are each to execute the task independently.

A task is also described by how long it normally takes to complete, whether it repeats and how quickly, and whether it starts automatically with the segment (implying that the assigned agents would become aware of the onset of the event phase on their own).

The purpose of a task can be stated as terminating the event phase of the segment, mitigating one or more risks or achieve gains within the segment's scope.

Related reference and policy documents can be attached to the task either by URL or by uploading them. Finally, the task can be given a brief, informal description.

When editing a “receive” or a “send”, the information needed, produced or to be shared is named and its elements are enumerated, and each assigned individual secrecy classifications and optional sourcing and handling instructions.

The mode of communication is given as either notification or by-request (push or pull).

If an information flow (information that's shared) is a notification, then it can be made a broadcast (information will be shared with all agents assigned to the target task) or a unicast (information will be shared with at least one agent assigned to the target task).

Keep in mind that an information flow that is a “send” to a task producing shared information is the same information flow that is a ”receive” to the task consuming the shared information. One is looking at one flow from its two ends.

An information flow is created by first adding an information need to a task (if necessary), then adding an information capability of the same name in another (also if necessary), and then connecting the need to the capability or vice-versa by setting the “from” or “to” field to the other task.

A communication delay is set either to zero (communication is to happen without delay) or to some finite time interval. For an information need, the delay represents the timeliness requirement for being notified or for getting a response to a request. For an information capability, the delay represents how quickly the information could be produced and communicated. For a sharing flow, the delay represents a “quality of service” commitment.

The communication channels over which the information is to flow can be set for both an information need, and for a notification information flow but only when viewed from the notified task. The same is true for a requested information flow when viewed from the task that responds to a request. Alternate communication channels are listed in order of preference.

The communication channels over which the information is to flow can be set for both an information need, and for a notification information flow but only when viewed from the notified task. The same is true for a requested information flow when viewed from the task that responds to a request. Alternate communication channels are listed in order of preference.

When editing a “send” or “receive”, a planner can set whether sharing would normally trigger or terminate a task. A notification could trigger or terminate the receiving task. It could also terminate the sending task, if sending the notification concludes the task that produced the information. A request could trigger the task that produces the requested response.

Planners can attach documents to a “send” or “receive”. A document can be a reference (it documents), a mandating policy (it obligates sharing), a prohibiting policy (it prohibits sharing), a multiple policy (contains language that imposes obligations and restrictions) or it can be an image provided as an illustration (for example, a screenshot). If an information flow has a prohibiting policy attached to it then there can not be any implied sharing commitment.

As with any Plan elements, a planner can add issues to those detected about tasks or its “sends” or “receives”. Manually added issues can be removed.

Segment scoping

Planners can edit the scope, name and description of a segment. A segment's edit form is accessed by selecting “About segment” from the top-most “Show” menu.

This opens a Details view on the selected segment where it can be renamed and where the phase and event it references can be changed. As with any other kind of Plan element, reference and policy documents can be attached to a segment.

The scope of a segment is further defined by identifying and describing a set of goals of organizations that relate to the segment's event phase. Goals can be risks to be mitigated or gains to be achieved. These goals are expected to be realized by tasks in the segment. Risks can also be curtailed by tasks that terminate the event phase. To be considered useful, tasks that do not curtail nor mitigate a risk or do not make gains are expected to at least provide information that is directly or indirectly needed by tasks that do.

The Goals view of a segment lets planners define the goals of organizations that are in scope. Goals can be added and removed. A goal is for an organization either to mitigate a risk or to make a gain. A goal is given a type (picked from a taxonomy of risks or gains), an importance (minor, major, severe for risks, low, medium or high for gains), the identity of the organization (either an actual organization or a type of organization), and whether the goal goes away when the event phase covered by the segment terminates.

The tasks that mitigate a given risk or achieve a goal in scope can be listed, sorted and navigated to.

The Map view locates on a Google® map all tasks in the segment that specify geo-locatable locations. The geo-mapping of elements is available throughout Channels.

Editing and analyzing entities

A new entity (an agent, role, organization, place, event, phase or communication medium) is automatically created whenever it is first referenced by name. Channels requires that all entities of a given kind be assigned unique names (John Smith, John Smith (1), John Smith (2) etc.).

Recall that the definitions of Plan elements can reference actual entities (ACME Corporation, Peter Sanchez, Morristown NJ) and types of entities (a corporation, a certified emergency technician, a city). An entity can only be tagged with entity types of its own kind (Warehouse A can be tagged as a building but not as a senior executive). Both actual entities and entity types can be edited. An editable and analytical view of an actual entity or type of entity can be opened from wherever it is referenced.

Keep in mind that events, roles and communication media in a Plan are always types. There is no actual event in a Plan because events are always anticipated. There is no such thing as an “actual role” because roles are always concepts. Communication media are also always types because a Plan only cares about the class of a communication medium.

The Details view of an entity enables a planner to change the entity's name, description, and types.

Each kind of entity has additional, specific attributes that can also be set. For example, a planner editing an actual organization can define its mission, and set its location, contact information, and parent organization. A planner can enforce that any sharing commitment by the organization with another be covered by an agreement, and that there be at least one known agent for each referenced role assigned by the organization. A planner edits the organization's employment roster of agents by either confirming employment implied by task in segments and assigning titles. A planner can also directly add new employees with their titles, roles and jurisdictions.

Cross-organizational sharing commitments are inferred from information flows in the Plan. A sharing agreement may or may not exist for a sharing commitment that crosses organizational boundaries. An unconfirmed agreement for a cross-organizational sharing commitment is an issue if such agreements are required for the organization. When viewing an organization, a planner can see the sharing commitments the organization makes with other organizations and confirm whether each commitment is covered or not by an agreement.

In the Details view of any kind of entity, references to the entity are listed as a filterable and paged index. Issues about the entity, if any, are enumerated at the bottom of the view.

In the Details view of an actual agent (as opposed to a type of agent), the agent, if it is a person and not a system, can be related to a user. The user, if not a planner, will be able to request to view his or her playbook (the agent's playbook) from the production version of the Plan. The agent can be declared to be a person or system, and given contact information (communication media with addresses where applicable) as well as clearances (secrecy classifications).

When editing a place from within the Details view, a planner can identify what other place, if any, contains it. A planner can also give the place a street address, postal code and a geoname. If the geoname is ambiguous, more specific geonames are offered for selection.

When editing a communication medium, a planner can set the transmission mode to either unicast (one -to-one), multicast (one-to-many) or broadcast (one-to-any), unless the mode is already implied by another medium this one is tagged with (to make it a special case of the tagging medium). The address pattern is a “regular expression”1 that constrains the form an “address” can take for this medium. A planner can identify the media, if any, this one interfaces with as it carries out transmissions from sender to receiver. Here again, some of the interfaced media might be implied from another medium this one is tagged with. Finally, a planner can describe the security of the communication medium in terms of security classifications. Qualifying a medium as “US Gov/Top Secret” means that Top Secret information can be transmitted securely over this medium.

The Network view is available on agents, roles, and organizations. It displays a network graph depicting how the entity is connected by information flows to other entities of the same kind. This analytical view allows a planner to filter the information flows to, say, show only the ones from tasks executed by a specific role. The network graph is navigable. The list of flows can be further filtered, sorted and paged. Listed flows can also be navigated to.

The Issues view collects and filters issues that are about an entity or that are related to it. Waived issues can be included in or excluded from the sortable, page-able list of issues.

Surveys, whether ongoing, closed, or to-be-created, are accessible from each listed issue. Only planners can launch or close surveys. A survey gathers the opinions of planners and non-planners alike on how a particular issue should be remediated.

Acquiring and releasing edit locks

More than one planner can concurrently work on a Plan. To prevent “collisions” where two planners are attempting to modify the same Plan element at the same time, only one planner is permitted to modify any given task, flow or entity at a time.

When a planner opens an edit form on a Plan element, Channels attempts to grab an edit lock on behalf of the planner for that Plan element. If another planner already owns the lock, the form is shown to the first planner with the editable fields disabled.

A lock on a task, flow or entity owned by a planner is released when the planner collapses or closes its edit form. An inactive or logged out planner eventually loses all acquired locks.

A planner waiting for edit locks is notified when the locks are released. The planner is then instructed to refresh the editor in order to acquire the locks and enable disabled edit forms.

Viewing task assignments and sharing commitments

Recall that tasks are defined in terms of agents, roles, places, and organizations, and that these entities can be actual or types. A task specification may reference entity types or omit to identify by name an actual agent, role, jurisdiction, etc. so as to encompass more than one possible agent as executor of the task. A task thus defined could be assigned to more than one agent.

A planner can look up the list of agents that are assigned to the task (the task assignments).

Agents are implicitly bound by sharing commitments when they are assigned to tasks that are sources of information flows. A planner can look up the list of sharing commitments inferred by an information flow.

Assessing the impacts of failure

Planners can assess what might happen should someone fail at an assigned task or should a commitment to share information be broken. The consequences could be a number of “downstream” tasks failing to make gains or mitigate or eliminate risks as planned.

For the sake of analysis, it is assumed that a failure is always complete; a failed task does not produce any information that may be required by others, and a failed information flow does not transfer any information whatsoever.

A failure can cause a cascade of task failures if tasks directly or indirectly depend critically on or are triggered by information that fails to be produced or to be shared.

Not all sharing failures matter. If an information flow is not critical to a task or does not trigger it, then its failure does not propagate to other tasks. Furthermore, if the target of a flow can get the same information via another, non-failing flow, then the flow's failure does not matter. Finally, if information produced by an agent is necessarily consumed by the same agent then, clearly, failure to share is meaningless.

Failure analysis only concerns itself with the failure of useful tasks, that is, tasks that either achieve stated goals or produce information that triggers or is critical to other tasks that are themselves useful. (Useless tasks are flagged as issues.) Note that a task will terminate a risk if it causes a segment to end (or, more to the point, if it causes the event phase the segment is about to end) and if the risk is said to terminate with the segment.

A planner does “what-if”, failure analysis by selecting a task or by selecting an information flow (“send” or “receive”) and then looking at the consequences should it fail. The planner can take a pessimistic outlook and assume that all redundant information flows would also fail, no matter what.

The result of the analysis is a cascade of failures visualized as a directed graph of information flows, tasks, gains and risks. Risks that would otherwise be mitigated or terminated by failed tasks are shown as color-coded flags. Orange is for severe risks, yellow for major risks and blue for minor risks. Risks that would normally be mitigated by successful tasks are shown connected to these tasks. Risks that would be terminated, should a segment end, are shown connected to red stop lights. Each stop light is itself connected to a failed task that would normally end the segment.

Missed gains are shown as stars and are similarly color-coded by importance, with orange for high, yellow for medium and blue for low.

In addition to the visual map, a tabular list of failed, goal-oriented tasks (the failures that matter) is compiled and presented. The table is paginated and the columns (failing task, segment, impact, goal, magnitude of the loss, and organization impacted) can be sorted.

Failure analysis crosses segment boundaries. A failed task in one segment can cause tasks to fail in other segments. Tasks, flows and goals in segments other than the selected segment are grouped in blue boxes.

Actions

Planners can create and remove segments; add, cut, copy and paste tasks, needs, capabilities and flows, connect tasks together with information flows or disconnect them. These actions are all available from “action” menus associated with editable Plan elements.

High-level actions can also be taken whenever applicable. They include “disintermediate” which removes a task that acts as a middleman in a flow, “intermediate” which adds a middleman task to a flow, and “satisfy needs” which automatically connects information needs of a task to matching but untapped information capabilities from other tasks.

Most actions can be undone and redone up and until a planner takes an action that conflicts with the undoing or redoing.

Adding and building a segment

When a planner creates a new segment, it is given a default name and a scope.

It is also initially populated with a single, unspecified task. A task can not be removed if it is the last task in a segment.

A planner would typically first name and describe the segment, set its scope (identify an event phase and goals to be achieved), and edit the lone task. Then, additional tasks would be added, together with their information needs and capabilities. These tasks would be connected via information flows with other tasks in the selected segment or other segments.

Issue detection and management

Channels automatically detects more than 60 different kinds of issues. Planners can manually add issues to Plan elements, describe these issues and suggest remediations. Manually added issues can later be removed.

Issue detection and remediation can provide guidance to planners as to what to do next when building a Plan. For example, a newly created segment with its lone task will immediately cause the detection of incompleteness issues. Resolving these incompleteness issues drives the planning process forward.

Some detected issues can be waived by planners if they are considered moot. When waiving an issue of a certain kind about a Plan element, all issues of this kind are also waived for the same element.

Connecting tasks with sharing flows

After adding a task, a planner may want to give it one or more information needs, and, for each need, name the information required and list the elements of information.

When a planner names an information, Channels suggests a list of already used information names that look like or mean something similar to the name being entered. This “semantic proximity”-based, auto-suggest capability leverages the WordNet lexical database and is available throughout Channels.

A planner may want to add an information capability to a task (shareable information produced by the specified task) and then connect it to a matching information need from another task in the selected segment or in another segment.

Connecting a capability with a need is done by selecting a task with a matching need from the “From” drop-down menu in the flow form (and vice-versa). Connecting a need with a capability creates a new information flow but does not remove the connected tasks' capability and need.

When a task has unsatisfied information needs (arrows to red dots), a planner may want to apply the “Satisfy needs” action. This automatically creates information flows with the other tasks in the Plan that can share the needed information.

Plan definition and analytics

A planner selects the item “About Plan” in the top “Show” menu to access views about the entire Plan.

The default Plan view is the Details view. Here, planners can change the name of a Plan, add phases and remove unreferenced phases. Planners can also add or view issues about the Plan as a whole.

In the All events view, a planner can add events to the scope of the Plan. A planner can mark an event as an incident if it is caused “outside” of the Plan and not (always) as an outcome of a planned task. There is at least one event defined in a Plan.

In the Secrecy classification view, a planner can define secrecy classifications that are then used to qualify elements of information and to set individual clearances. References to a classification can be listed. Unreferenced classifications can be removed.

In the All organizations view, a planner can identify the organizations that are in the default context of the Plan.

In the All segments view, planners can look at a map of all segments in the Plan. The segments can be grouped by phase or by event. Planners can browse all cross-segment sharing flows as well as all tasks in one segment that start or terminate another segment.

In the Who's who view, planners can browse a directory of all agents involved in the Plan.

In the All issues view, planners can sort, filter and browse all issues currently detected or manually added within the Plan. Surveys on individual issues can be created or accessed from this view.

In the Bibliography view, planners browse all the documents that are attached to Plan elements anywhere in the Plan.

In the Index view, planners can find, filter, sort and navigate to all entities referenced in the Plan.

The Evaluation view gives a summary evaluation of the Plan in terms of categories of issues detected and added by planners.

Finally, the Version view is where planners vote to move the development version of the Plan into production. A unanimous vote is required. Moving a development version into production creates a development version from a copy of the new production version. Planners can be contacted by email from within the view to prompt them to vote.

Secrecy, security and clearances

Information classification, the security of communications and the clearances of agents can be important elements of information sharing planning in domains of human endeavor that require the compartmentalization of information. Planners can introduce one or more classification systems to a Plan.

Each classification system has a unique name and consists of a list of classifications (secrecy levels) ranked from highest to lowest.

Agents can be given clearances. Clearances are expressed as classifications (no more than one per classification system).

How secure a communication media type is is defined by the highest classification levels of the information it can transmit without infringing security requirements.

Elements of information shared or to be shared can each be given classification levels as well as indications of the source and special handling instructions.

Secrecy and security issues are automatically detected such as:

  • Classified information transmitted over an insufficiently secure channel

  • A commitment to share classified information with an agent who has insufficient clearances

  • Receiving classified information and then disseminating it with reduced classification

Surveying planners and participants

Planners can survey other planners and participants assigned to the current Plan about their preferred remediations for issues that are detected by Channels or that are manually entered by planners.

Channels interfaces with the SurveyGizmo® web service to create, launch, and retire surveys, and to collect responses.

Publishing information sharing procedures

Channels produces on demand an information sharing procedures report and individualized information sharing playbooks for the current Plan. The report is accessible to planners but not to participants.

The report organizes by segments and agents the Plan's task assignments, sharing commitments and related issues, and presents them in a single HTML document suitable for browsing and printing.

The Plan map and the information flow maps shown in the report are reduced to fit. Full-sized Plan map and flow maps can be retrieved as PNGs.

From the header of the report, a planner can narrow the focus to a single segment and/or agent. The inclusion of issues can also be turned on or off.

A participant can only access the information sharing procedures for the agent he or she represents, and only from the production version of a Plan.

Logging out

A planner logs out to terminate a Plan editing session. Any edit locks still held at that time are automatically released. The user is then presented with a login page.

The meaning of information in ISP

Tasks and flows are defined in terms of information needed, uncovered and shared. Information is named and is broken into atomic elements. Each element of information is named, can be given security classifications and can be annotated with a description and handling instructions.

Figure 1.31. Elements of information


Elements of information sent from a task can be described as the result of transforming elements received in the execution of the task. An element of information (EOI) sent can be the same as an EOI received, can be an EOI received but renamed, or an aggregation of EOIs received (for example a count, a synthesis etc.).

An element of information is considered atomic for the purposes of a plan; it is the indivisible unit of transmission and transformation. Once aggregated, an EOI can not be "split apart".

The notion of the plurality of an element is left undefined. For example, if "suspicious activity report" information contains the element "suspect profile" then it is possible that there will be more than one suspect profiles in a report.

When analyzing flows, Channels is not concerned with what the information actually means. Channels has no domain knowledge to infer from. Rather, it cares about whether or not two flows are about the same information, and whether an element of information is the same as, or is the transformation of, another element of information.

If the information in two flows have the same name, it does not necessarily mean that they represent, conceptually, the same information. Two tasks performed respectively by a EOC commander and a corporate CEO might both need a "situation assessment" but they would likely not need the same kind of situation assessment with the same elements of information (even if the elements themselves have identical names).

In Channels, identically named flows, needs and capabilities (uppercase vs lowercase and extra spaces are ignored) are explicitly meant to represent the same kind of information (they are synonymous) if and only if they connect to the same tasks (one flow may be a "send" and the other one a "receive"). Similarly, identically named elements of information are meant to represent the same atom of information if only if they compose synonymous information.

Agents executing a task receive and send information. The information they receive and send may have different names an yet have common elements of information. For example, the "suspect profile" element of a sent "suspicious activity report" information may be the same as the "suspect description" element of a received "security tip", and it may be the same as the "suspect profile" of received "BOLO" information.

Alternately, an element sent may aggregate elements received. Aggregation combines many elements into a single element, with loss of data. For example, a "suspects count" element aggregates "suspect profile" elements into a single number, losing the details of the individual profiles. Counting is only one of many possible kinds of aggregation. A "threat level", for example, might aggregate a great deal of different elements into a high vs low rating.

In order to track the dissemination of information, it necessary to indicate when sent elements are the result of renaming or are aggregating received elements. Channels need only differentiate between two kinds of element transformation in order to track the dissemination of information: renaming and aggregation. The description field of an element of information can be used to provide more details about the specific nature of an aggregation.

Issue detection takes element renaming and aggregation into account. For example, while it may be acceptable to declassify aggregated elements, it is not acceptable to declassify a renamed, classified element.

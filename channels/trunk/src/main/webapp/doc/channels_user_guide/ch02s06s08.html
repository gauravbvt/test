Tasks, receives and sends panels

The task, receives and sends panels at the bottom of the plan editor show summaries of the selected task (there is always a task that is selected) together with a list of summaries of its "receives" (the task's information needs and the information flows to it) and a list of its "sends" (the task's information capabilities and the information flows from it). Each summary can be expanded into a form for editing (if the development version of the plan was the one selected in the home page).

Each of the summaries is prefixed with a colored bullet. The color of the bullet indicates how important a task or a flow is. The color indicates the calculated impact of the task or the flow failing completely. Blue indicates a minor impact, yellow is major, orange is severe and red is extreme. Gray bullets are used when failure impact does not apply (information needs and capabilities).

Clicking on the summary of a task or flow expands it by opening its edit form. Channels automatically adapts the width of task and flow forms to ease editing. Clicking on the summary closes the form.

Note that, with very few exceptions, any and all fields in any form in Channels can be left blank.

Editing a task

A task is defined by naming it, categorizing it, tagging it and flagging, or not, as operational and/or prohibited. (Remember, everything is optional.) It is further defined by stating where it is to be executed, by whom, in what role within what jurisdiction, and on behalf of what organization. Any of the location, agent, jurisdiction, role, and organization (entities) that specify a task can be left unnamed so as to imply that more than one agent can be assigned to the task. The names of entities can be indicated to refer to actual entities (Morristown, NJ, John Doe, ACME etc.) or to types of entities (a city, a certified technician, a corporation etc.).

The execution of a task can be set to be “as a team” to indicate that all assigned agents are to work together when carrying out the task. By default, assignees are each to execute the task independently.

A task is also described by how long it normally takes to complete, whether it repeats and how quickly, and whether it starts automatically with the segment (implying that the assigned agents would become aware on their own of the onset of the segment's event phase).

The purpose of a task can be stated as causing an event, terminating the event phase addressed by the segment, mitigating one or more risks or achieving gains within the segment's scope.

Reference and policy documents can be attached to the task either by URL or by uploading them. Finally, the task can be given a brief, informal set of instructions.

As for any plan element, a planner can add issues to those automatically detected about the task. Issues added by a planner can be later removed by any other planner. Issues detected by Channels can not be removed but might be waived.

The task's "show" menu opens various "views" on the task, namely:

  • The details of the task as an edit form - a shortcut is to click on the task's summary

  • The assignments panel - all agents that are assigned to the task

  • Override panel - showing tasks that override or are overridden by this task (the menu item is not shown if there are no overrides)

  • The dissemination panel - analysis of element of information dissemination to and from the task

  • Entity profile panels - for all entities referenced in the definition of the task

Figure 2.53. A task's "show" menu


The task actions menu becomes available when the task is expanded into its edit form (and the planner has acquired exclusive editing rights). Actions that do not apply are grayed out.

The actions available from the task actions menu are:

  • Undo the planner's last change to any plan element, if applicable

  • Redo the planner's last undone change, if applicable

  • Send a message about the task

  • Take a copy of the task

  • Cut (copy and delete) the task

  • Paste a copied attachment to the task's documentation, if applicable

  • Paste the copied task, if any - apply the copied task's definition to this task

  • Create a new task in this segment that's a copy of this one

  • Connect this task with flows from other tasks that can satisfy (partially or completely) the info needs of this task

  • Disintermediate this task - directly connect tasks that use this one as a "middleman" for information sharing, bypassing this task

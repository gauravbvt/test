Editing a "send" or "receive"

The first step in defining either an information need, an information capability or an information flow is to name the information, identify its intent (command, alarm, announcement, report, or feedback), and list the elements of information.

The mode of communication is given as either notification or by-request (push or pull).

Planners can specify if the information would be shared only in case to task failure ("sends" only).

If an information flow (information that's shared) is a notification, then it can be made a broadcast (information will be shared with all agents assigned to the target task) or a unicast (information will be shared with at least one agent assigned to the target task).

Keep in mind that an information flow that is a “send” to its source task is a ”receive” to its target task. One is looking at one flow from either of its two ends.

An information flow can be created by first adding an information need to a task (if necessary), then adding an information capability of the same name in another (also if necessary), and then connecting the need to the capability or vice-versa by setting the “from” or “to” field to the other task. Another way to create an information flow is to start from an information need or capability of a task and then name the task at the other end of the flow. If a task of that name does not exist, a new one is created and connected.

When specifying or determining the source or target of a flow, a planner can restrict the sources and targets to be within the same overall organization, within the same organization or within the same location. Adding a restriction on a sharing flow may filter out sharing commitments that would otherwise be implied by the flow (e.g. by dropping any sharing commitment that crosses organizational boundaries).

A communication delay is set either to zero (communication is to happen without delay) or to some finite time interval. For an information need, the delay represents the timeliness requirement for being notified (maximum delay from time of availability), or for getting a response to a request (maximum time elapsed from the request). For an information capability, the delay represents how quickly the information could be produced and communicated. For an information flow, the delay represents a “quality of service” commitment.

The communication channels over which the information is to flow can be set for both an information need, and for a notification information flow but only when viewed from the notified task. The same is true for a requested information flow when viewed from the task that responds to a request. Alternate communication channels are listed in order of preference.

When editing a “send” or “receive”, a planner can set whether sharing would normally trigger or terminate a task. A notification could trigger or terminate the receiving task. It could also terminate the sending task, if sending the notification concludes the task that produced the information. A request could trigger the task that produces a reply.

Planners can attach documents to a “send” or “receive”. A document can be a reference, a mandating policy, a prohibiting policy, an unspecified policy (contains language that may impose both obligations and restrictions) or it can be an image provided as an illustration (for example, a screenshot). Note that attaching a prohibiting policy to an information flow does not prohibit the generation of information sharing commitments.

Instructions can be given as text in the definition of "sends" and viewed in the definitions of received information flows.

Information needed, shareable or to-be shared can be decomposed into named elements. Each element of information can be described and assigned individual secrecy classifications as well as handling instructions. Planners can indicate whether and how a "sent" element of information is related to a received element of information (is the same one but renamed, is it an aggregation of many received elements?).

An element sent is implicitly the same as one received if both are identically named and both compose identically named incoming and outgoing information. If an element sent is in effect the same as one received but they have different names or compose information with different name, then planners need to indicate that the element sent renames the element received. If an element sent is a compilation/summation of multiple elements received, planners can identify the received elements that are aggregated into a sent element. Doing so enables Channels to track the planned dissemination of information.

To reduce the drudgery of defining elements of information, the list can be auto-populated by clicking on the magic wand icon. Clicking on the classifications "link" icon forces all elements to have the same classification. Clicking again removes the constraint.

The elements of information of an information need do not have secrecy classifications, descriptions, special handling instructions or transformation attributes. They do however have time-sensitivity attributes to indicate which of the elements are subject to the timing requirements of the information need they define.

The "more" menu of a "send" or "receive" combines access to "views" and to "actions". The menu in a collapsed "send" or "receive" differs from the menu when the edit form is expanded and the planner gains exclusive rights to make changes.

The menu of collapsed information need allows the planner to:

  • Show or hide (expand or collapse) the edit form

  • Send a message about the information need

  • Undo the planner's last change to any plan element, if applicable

  • Redo the planner's last undone change, if applicable

  • Make a copy of the information need

The menu for an expanded information need additionally allows the planner to:

  • Edit the elements of an information need

  • Add a new, ad hoc issue about the information need

  • Add a copied attachment to the information need's documentation, if applicable

  • Remove the information need

Figure 2.60. Information need menus for collapsed and expanded edit forms


The menu of collapsed information capability allows the planner to:

  • Show or hide (expand or collapse) the edit form

  • Send a message about the information capability

  • Undo the planner's last change to any plan element, if applicable

  • Redo the planner's last undone change, if applicable

  • Make a copy of the information capability

The menu for an expanded information capability additionally allows the planner to:

  • Edit the information capability's elements

  • Add a new, ad hoc issue about the information capability

  • Add a copied attachment to the information capability documentation, if applicable

  • Remove the information capability

Figure 2.61. Information capability menus for collapsed and expanded edit forms


The menu of collapsed information flow allows the planner to:

  • Show or hide (expand or collapse) the edit form

  • Send a message about the information flow

  • View the information sharing commitments computed for this flow

  • View the failure impact analysis for this flow

  • View the information dissemination analysis for this flow

  • Undo the planner's last change to any plan element, if applicable

  • Redo the planner's last undone change, if applicable

  • Make a copy of the information flow

The menu for an expanded information flow additionally allows the planner to:

  • Edit the information flow elements

  • Make a copy of this flow

  • Add the information need (if flow is receive) or capability (if flow is send) implied by this flow (duplicate flow)

  • Add a new, ad hoc issue about the information flow

  • Add a copied attachment to the information flow documentation, if applicable

  • Remove the information flow

  • Break the flow into two flows by inserting an undefined, intermediate task in the flow

  • Break up the flow by replacing it, if not redundant, by the implied information capability to the source task plus, also if not redundant, by the implied information need to the target task

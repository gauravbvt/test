Activities

An ISP Method activity is carried out by an IS planner either to correct a mistake in the definition of an IS plan, bring the plan closer to completion, or make it more robust.

An activity is composed of a number of steps, all of which may be optional, often to be carried out in no particular order. Most activities can be undone (for example, a task added to the IS plan can be later removed). Prerequisites for an activity, if there are any, are usually few and commonsensical. For example, a task in a segment can not be specified unless the segment is already defined.

Note that for each "Add ..." activity there is usually an implied and related "Remove ..." and "Modify ..." activity. For example, "Add task" implies "Remove task" and "Modify task".

Prior to applying any activity, it is assumed that:

  • an IS plan has been created and given a unique identifier (the identifier is not the name of the plan; it is universally unique and can not be changed once the plan has been created)

  • users have been authorized as planners

  • a default segment is created for a default phase of a default incident event (Channels does it automatically)

Activities fall under scoping a plan, specifying information sharing procedures, profiling entities, or improving a plan.

Scoping a plan

The activities that modify the scope of an IS plan are:

  • Localize the IS plan

  • Identify an event as incident

  • Identify a planned phase

  • Identify a participating organization

  • Identify a participating agent

  • Add a segment

  • Add an organizational goal

  • Add a secrecy classification system

Localize the IS plan

Localizing an IS plan sets the locale or type of locale where the plan applies. This allows other places to be defined relative to the plan's locale thus making easy to "move the plan" to a different locale (for example, from Philadelphia, PA to New York, NY). Localizing a plan is optional.

Table 3.1. Scoping: Localize the IS plan
Prerequisites:

  • The plan was created
Steps:

  • Set the locale of the plan either as an actual place or as a type of place.
Outcomes:

  • Place types (other than the plan's locale and places it is defined by) can be defined relative to the plan's locale.

Figure 3.23. Setting the plan's locale to a type of place


A plan's locale need not be set. If it is set, then place types within the plan can be defined to be within or containing the plan's locale (whatever it may be). Of course, a place identified as the plan's locale can not be defined relative to the plan's locale, either directly or indirectly through the definition of its place categories, containing places, or contained places. (See "Profiling a place").

Identify an event as incident

The scope of an IS plan is largely set by identifying the types of anticipated events that are of concern. An event is considered an "incident" if it is unplanned (no task within the plan is expected to cause it). An IS plan always has at least one incident event within its scope. When an IS plan is created, an "unnamed" event is put in its list of incidents.

Table 3.2. Scoping: Identify an event as incident
Prerequisites:

  • The plan was created.
Steps:

  • Name a new event and mark it as an incident.

  • Or, mark an event already referenced in the plan as an incident.
Outcomes:

  • A phase (in scope) of this event can now be the focus of a plan segment.

Figure 3.24. Events can be marked as incidents and new ones added


Identify a planned phase

When an IS plan is created, Channels automatically defines a "co-event" phase and calls it "Responding". If planners want to also plan for the "before" or "after" of events, they need to add them to the scope of the plan by naming them in the Details view of the plan definition panel. Planners can also rename the default "Responding" phase to anything they want. A phase that is not referenced anywhere, other than in the plan's Details view, can be removed from the scope.

Table 3.3. Scoping: Identify a planned phase
Prerequisites:

  • The plan was created.
Steps:

  • Name a new phase

  • Set the phase as pre-, co- or post-event

  • Describe the phase (optional)
Outcomes:

  • This phase of an event (in scope) can now be the focus of a segment.

Figure 3.25. A new phase is added to the scope by naming it in the details of a plan


Figure 3.26. The phase is defined as "co-event" and described


Note that categorizing a phase (always a type) in terms of another has not practical effect.

Identify a participating organization

An actual organization (as opposed to a type of organization) is implicitly added to the scope of an IS plan any time it is referenced by name in the definition of a task or of some other plan element. A planner can also make an (actual) organization part of the scope of a plan, by adding it directly to the list of all organizations in scope.

Table 3.4. Scoping: Identify a participating organization
Prerequisites:

  • The plan was created.
Steps:

  • Identify an actual organization in a task definition and add the expectation that it will be assigned tasks

  • Or, add an actual organization explicitly to the scope
Outcomes:

  • The organization remains durably in the scope of the plan (even if it not referenced in any task definition).

  • An issue is raised if no member of the organization is assigned at least one task.

  • The organization can be profiled.

  • Participating agents for this organization can be identified.

Figure 3.27. Adding implicitly an actual organization to the scope


Figure 3.28. Putting an organization permanently in scope by setting expectation of participation


Figure 3.29. Directly adding an organization permanently to the scope


Organizations added to the scope other than by directly adding them to the list of participating organizations will be removed from the scope if they are no longer referenced within the plan. By adding an organization to the list, it remains in scope even if it is not referenced by name in any task definition (or in another organization's profile). An organization already referenced in the plan can be added durably to the scope by "adding expectations" that it is being assigned tasks. A planner can later remove such expectations.

Identify a participating agent

The scope of an IS plan includes not only participating organizations but also participating agents, that is, agents as role-playing members of organizations that can be assigned to tasks. Agents "employed by organizations" are added to the scope implicitly from the definitions of tasks or explicitly by defining the "jobs" of participating organizations that are relevant to the plan.

Since it is a best practice not to directly identify the assignees of tasks but rather to specify them via their roles in types of organizations, the preferred approach for identifying participating agents is to list them with their jobs in the profiles of participating organizations.

An agent is further identified by associating one or more users to it. The meaning of associating a user to an (actual) agent depends on whether the agent is designated as an archetype, a placeholder or, the default, as a normal agent.

An archetype agent captures the notion of a group of anonymous and indistinguishable agents (same contact info, same roles in the same organizations). An "ACME Call Center Agent" would be an archetype playing the role of "Call Center Agent" for the organization "ACME" and with some 1-800 number as communication channel. Any number of Channels users can be associated with an archetype. These users all receive the IS guidelines of their associated archetype.

A placeholder agent is an agent whose identity is to be given via their associations with users. A placeholder agent can be given default contact information that the associated user(s) can override and add to. A placeholder agent can be restricted to only one associated user or can be open to associations with many. A user associated with a placeholder agent gets the IS guidelines of that agent, with the user's contact info substitutions/additions.

The default kind of actual agent directly identifies the person or system and gives its contact information as part of the profile. Only one user can be associated by planners with a normal, actual agent. The full name of the user and of the agent are expected to match (an issue is raised if not). The associated user gets the agent's IS guidelines. It is an ISP best practice to define agents as much as possible either as archetypes or as placeholders.

Table 3.5. Scoping: Identify a participating agent
Prerequisites:

  • A participating organization was identified.
Steps:

  • Identify an actual agent in a task definition together with role and actual organization (not recommended)

  • Or, add the agent with job definition to the profile of the participating organization the agent is a member of (preferred).

  • Set the agent as either normal (the default), an archetype or a placeholder (See Profiling Activities - Profile an agent)

  • Associate one or more users with the agents
Outcomes:

  • The agent is automatically assigned to tasks where it matches assignee specifications.

  • An issue is raised if a participating agent is not assigned to any task.

  • The agent can be profiled.

  • Users associated with agents can retrieve their IS guidelines.

Figure 3.31. Confirming an otherwise implied agent's job


Figure 3.32. Adding a job from within an organization's profile

Add a plan segment

The information sharing model of an IS plan is composed of segments, each focusing on one phase of one event, possibly in the context of other past or current events. Each segment defines a list of organization goals to be achieved via flow-connected tasks. Adding a segment contributes to setting the scope of a plan because doing so is a pre-requisite to adding organization goals that contribute to the scope.

Table 3.6. Scoping: Add a plan segment
Prerequisites:

  • An IS plan is created
Steps:

  • Add a new segment to the plan.

  • Name and optionally describe the segment.

  • Set the phase of the event in focus and rate its required/"threshold" level (minor, major, high, very high).

  • Set the past or present events in context.
Outcomes:

  • Organizational goals can be added to the segment.

  • Tasks and flows can be added to the segment.

Figure 3.35. Adding a new segment


Figure 3.36. Identifying phase and event in focus, rating it, and adding contextual events


When an IS plan is created, it initially contains an "untitled" segment focusing on the "Responding" phase of the "unnamed" event. The segment has a single, undefined task and no organizational goals.

A segment can be removed in its entirety, after confirmation. The operation can be undone.

A segment identifies any number of evens as context to the event phase in focus. A segment's context represent what events have occurred or are already occurring when the event phase in focus starts. Events can also be removed from the context.

Add an organizational goal

Adding a goal to a segment augments the scope of an IS plan. An organizational goal is either a risk to be mitigated or a gain to be made by an organization (or type of) within the segment where it is listed.

A goal is categorized by choosing from a fixed list of risk or gain types (based on actuarial risk management taxonomies), graded for severity/magnitude, and briefly described with free-form text. Planners can also indicate whether a goal is implicitly achieved with the termination of the segment's event phase (for example, the risk of property damage is terminated when responding to a fire is over, but the risk of litigation is not).

Table 3.7. Scoping: Add an organization goal
Prerequisites:

  • The segment meant to achieve this goal was added.
Steps:

  • Add a risk or gain to be achieved and categorize it.

  • Rate the severity of the risk or magnitude of the gain.

  • Name the actual or type of organization that needs to achieve the goal.

  • Set whether the goal is implicitly achieved when the event phase in focus for the segment terminates.

  • Briefly describe the goal (optional).
Outcomes:

  • Tasks can be defined as accomplishing this goal.

Figure 3.37. Adding a goal to a segment, looking up the tasks that references a goal


A goal can only be removed from the list if it is no referenced by any task within the segment. References to the goal are shown in the Goals view of the segment definition panel.

Add a secrecy classification system

A secrecy classification system is a named ordered list of classification names (for example, US Government: sensitive, confidential, secret, top secret). Secrecy classifications are used to set agent clearances, communication channels' security and the classification levels of elements of information. When an IS plan is created, it defines no secrecy classification system. Planners must first define a secrecy classification systems before they can be applied.

Figure 3.38. Adding a secrecy classification system and level


A secrecy classification level can be removed from a secrecy classification system if it is not used anywhere within the plan. Planners can see where a classification level is being used.

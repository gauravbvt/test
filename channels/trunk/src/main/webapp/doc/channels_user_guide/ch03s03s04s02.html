Generalizing a task's definition

Sometimes the assignees of a task are specified too narrowly in the definition of the task, possibly because an organization is identified by name in the definition, or because the role given is too specific. It can also be the case that an IS plan contains multiple tasks, each with the same name but with different, narrowly defined assignee specifications, that could be replaced by a single task with more broadly specified assignees.

However, when two broadly defined tasks are connected by a flow, many of the implied information sharing commitments may be inappropriate. For example, assignees to both the source and target tasks of a flow may be from different organizations (organization types were used to specify task assignees). If the intent was for task assignees to communicate only with co-workers, then a planner must put in a restriction on the flow stating that sharing commitments are only between agents in the same organization. The alternative is to define multiple source and target tasks, each narrowly assigned to members of only one organization (but otherwise identically named pair-wise), and connect them with flows of the same name.

It is an ISP best practice to avoid as much as possible having multiple tasks of the same name in the same segment, except for purposes of overriding tasks and/or flows (see section on Overriding tasks and flows). Generalizing task assignee specifications combined with flow restrictions can greatly reduce the number of tasks and flows in an IS plan, thus making it more manageable and easier to grow.

Table 3.20. Improving a plan: Generalize a task's definition
Prerequisites:

  • The task was created and defined
Steps:

Apply any or all of these steps:

  • Remove a reference to an actual agent from the definition of a task or replace it by an agent type

  • Replace in the definition of a task a role, agent type, jurisdiction type, or organization type by another that categorizes it

  • Replace in the definition of a task a reference to an actual organization by an organization type

  • Replace in the definition of a task a reference to an actual place as the jurisdiction by a place type

  • Add a restriction to connecting flows to prevent the generation of unwanted sharing commitments

  • Remove from a segment tasks of the same name that are now subsumed by the generalized task definition
Outcomes:

  • Fewer tasks and flows specify the intended tasks assignments and information sharing commitments

Flows and tasks

Flows and the tasks they connect are the building blocks of the IS practices model in an IS plan. A task describes an action carried out at some location and for some purpose by one or more agents. Agents act to fulfill the responsibilities of the roles bestowed on them by organizations. The agents expected to carry out a task can be identified by any combination of name, role, jurisdiction and organization. The executor of a task may be identified only partially (for example, by role and organization only) so that more than one person or system will match the profile and get assigned to the task. According to a tasks' definition, those assigned to the task are expected to carry it out either as a team or on their own. The timing aspects of a task execution can be given, namely, how long the task takes to complete and how soon it repeats, if at all. A task might cause or prevent an event.

Tasks are considered useful if they:

  • directly achieve goals (mitigate risks or capture gains),

  • initiate or terminate (phases of) events,

  • or produce information needed by other useful tasks.

Tasks can be triggered or terminated by:

  • direct awareness of events,

  • receiving notifications,

  • receiving requests for information

An information flow links a task that produces information to a task that consumes it. A flow is characterized by:

  • whether the flow is a notification, a request or a reply

  • the information communicated, down to its elements, each with optional secrecy classifications and sourcing and handling instructions,

  • whether the situation (even phase and event context) in which the source task is being executed (e.g. responding to a suspected attack) is also to be communicated,

  • the intent of the flow (alarm, announcement, command, feedback or report)

  • whether the flow is conditional on task failure

  • the mode of communication (request vs notification, unicast vs multicast vs broadcast),

  • restrictions, if any, based on shared organization membership or location of the source and/or target of the flow,

  • a list of preferred transmission media (phone, face-to-face etc.),

  • the urgency (maximum delay from earliest availability if a notification, or maximum delay from request if a reply)

  • the impacts, if any, the communication may have (it might trigger or terminate a task),

  • instructions on how to carryout the task,

  • policies that mandate or prohibit sharing.

A task can have information needs (information agents require executing the task), information capabilities (information acquired by the execution of the task that can then be shared), or both. An information flow implicitly specifies an information need of the receiving task that is satisfied by an implied information capability of the sending task. Planners can explicitly define the information needs and capabilities of a task.

Explicitly specifying an information need of a task allows Channels to automatically verify if the need is satisfied (Are all elements supplied in a timely manner by incoming flows?) and to find information-capable tasks that could be connected by flows to satisfy an information need.

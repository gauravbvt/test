Profiling a place

Places are used in IS plans to

  • Give the locale of a plan (in terms of which other places can be defined)

  • Define the locality of events

  • Situate the execution of a task

  • Define the jurisdiction of an agent in a role assigned by an organization

  • Give the address of an organization

  • Define other places in relative terms

A place entity can either be a type of place or an actual place.

Table 3.14. Profiling: Profile a place
Prerequisites:

  • The place or type of place was named
Steps:

(All steps are optional and they can be taken in any order)

  • Change the name of the place

  • Describe the place

  • Tag the place

  • Categorize the place with other places (avoiding circular categorization)

  • Attach a document (picture, policy, or reference)

For actual places:

  • Another (actual) place that directly contains this one (e.g., "Room A-101" is in "Building A")

  • A street address

  • A postal code

  • An administrative, geographical area (for example, "Newark, New Jersey, United States") - disambiguated if necessary

For types of places:

  • Constraints on what places can contain this type of place (containment constraint) or be contained in this type of place (content constraint):

    • the plan's locale

    • the location of a given event

    • a given place (actual or type)
Outcomes:

  • Depending on its categorization, the place may or may not be compatible as a category for certain other places.

  • Assignments, and thus sharing commitments, may change for tasks that specify a jurisdiction.

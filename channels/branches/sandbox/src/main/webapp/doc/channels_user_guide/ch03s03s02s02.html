Adding a task's information need

The incoming flows of a task implicitly define satisfied needs for the task. By adding an explicit information need to a task, a planner defines an information requirement of the task that may or may not be satisfied by incoming flows. The planner also indicates that incoming flows partially satisfying or completing this information need are not necessarily the only acceptable sources. Channels looks at the information needs and information capabilities of tasks to uncover missed opportunities for information sharing (see "Satisfy all needs" in the Commands section).

Note that only the explicit information needs of a task are visible across segments and offered as candidate targets for cross-segment flows. (Software types might want to think of information needs and information capabilities as defining the "interfaces of a segment.")

Table 3.9. Scoping: Add a task's information need
Prerequisites:

  • The task was added.
Steps:

(All steps are optional and they can be taken in any order) Note: Tagging with information standard tags enables auto-populatio of standard elements of information.

  • Name the information

  • Categorize expected intent as one of the following:

    • report (summary of facts and conjectures: "Here's what we know...")

    • command (an injunction: "Do this!" "Stop that!")

    • announcement (declaration, communiqué, advertisement: “I-95 is congested”, “Spinach is safe”)

    • feedback (thumbs up or down, reward, punishment: "Objectives were met", "Report is incomplete")

    • alarm (signal of an unexpected change requiring outside assistance ("Distribution point is overwhelmed", “Levee is breached”)

  • Name the information's elements and indicate which ones are time-sensitive (i.e., are subject to timing requirements).

  • Indicate whether the needed information would be received as a notification or as an answer to a request.

  • Indicate what the impact on the task would be of receiving the needed information, as

    • useful to the task

    • critical to the task (the task may fail without it)

    • triggers the task

    • terminates the task

  • State a relationship restriction on the source agent, if any. Namely the source agent must be

    • from the same overall organization

    • from a different overall organization

    • from the (exact) same organization

    • from a different (specific) organization

    • in the same location

    • in the same location and organization

    • in a different location

    • from a supervisor

    • from self

    • from someone else

  • List the alternate transmission channels to be used, in order of preference.

  • Give time constraints on receiving the information

    • If the need is to be satisfied via notification, the time constraint gives the maximum delay from first availability to the task via dissemination of time-sensitive elements of information.

    • If the need is to be satisfied via reply, the time constraint gives the maximum delay between request and reply.
Outcomes:

  • Whether the information need is satisfied can be diagnosed.

  • Channels can check whether incoming flows of the same name conform to expectations (channels, timing, elements, etc.) expressed in the need.

  • Channels can automate the creation of flows that satisfy the need.

  • Channels can offer the task as a candidate target for the capability of a task from another segment.

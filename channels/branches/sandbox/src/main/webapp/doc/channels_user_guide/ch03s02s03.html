Moving an IS plan to completion

When is an IS plan complete? There is really no satisfactory answer to this question. Because one could always increase the scope of a plan, a more useful question would be: Given its current scope, is there something clearly missing from a plan? If the answer is no, then the plan can be deemed “complete.”

Knowing whether the scope of an IS plan is properly set requires knowing the domains of human endeavor involved and being able to identify the organizations that ought to participate in the plan. Absent this knowledge, software can only examine an IS plan to detect what is inherently missing from it.

Nothing is inherently missing from a plan if:

  • Each phase of each event in scope is addressed by at least one segment of the plan.

  • An event is not self-terminating; each segment addressing a co-event phase for that event has at least one task that terminates the segment.

  • Each segment has at least one defined goal.

  • Each goal listed in a segment is achieved by at least one task within that segment.

  • Each participating organization and agent within the scope of the plan is assigned at least one task.

  • Each task is assigned to at least one agent.

  • Each information sharing flow generates at least one information sharing commitment.

  • Each task is useful in that it directly achieves a goal, or directly or indirectly enables a task that does.

  • Each referenced role in an organization is assigned to at least one member agent (if required).

  • Each inter-organizational information sharing commitment is covered by a sharing agreement documented by a Memorandum of Understanding (if required).

The scope of an IS plan can be changed even after it is deemed complete. Tasks, flows, and entities that do not contribute to completing a plan are considered redundant. Redundant tasks, flows, or entities should be removed from a plan. If the scope of a plan is reduced, some elements of a plan may become redundant as a result.

The ISP Method proposes three distinct strategies for driving an IS plan toward completion:

  • Goal-driven strategy - Add goal-achieving tasks and work backward to satisfy their information needs.

  • Effect-driven strategy - Add tasks that execute at the onset of event phases and work forward by connecting tasks with outgoing flows.

  • Issue-driven strategy - Resolve or waive outstanding issues until none remains.

Planners usually use of all three strategies opportunistically when building an IS plan.

Goal-driven strategy

Figure 3.7. Goal-driven IS plan building


Planners look at each organizational goal attached to a phase of an anticipated event, add tasks that achieve it, and add the information needs of each of these tasks. They then add tasks that produce the needed information and connect the tasks with information flows. This process continues until all goals are addressed by tasks, and all task information needs are satisfied.

Systematic use of a goal-driven strategy guarantees that all tasks and flows, and consequently the IS procedures generated from them, are within the scope of the IS plan, because all tasks and flows contribute directly or indirectly to addressing identified goals. However, this strategy alone does not guarantee that tasks triggered by the start of event phases will be included. In other words, this "working from the end" strategy does not guarantee that it will reach all starting points.

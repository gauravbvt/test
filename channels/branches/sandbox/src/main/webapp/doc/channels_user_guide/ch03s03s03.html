Profiling entities

In ISP entities encompass the "static elements" in IS plans that can be unambiguously identified and thus uniquely named. ISP entities are events, phases (of events), agents (people or systems), organizations, roles, places, and communication media (phone, email etc.) Tasks and flows are not entities because they are neither static (they are what agents do) nor are they uniquely named.

Entities are referenced by name in the definitions of a plan, plan segments, tasks, flows and of other entities. An entity is created the first time it is named and is removed when no longer referenced (immediately if its definition is empty).

Entities can be "actual" or "types". An "actual" entity represents a physical reality whereas a "type of entity" represents a category of entity. "John Doe" would be an actual agent whereas "Adult" would be a type of agent. "The United Nations" is an actual organization whereas "International Governing Body" is a type of organization. And so on. Some kinds of entities can only be types in an IS plan, namely events, phases, roles and communication media. ISP does not deal with actual events, only with types of anticipated events. Roles, phases and communication media are inherently categories.

Planners can define (profile) an entity as soon as it is first referenced by giving values to some or all of the entity's attributes. All entities, be they types or actual, have a few attributes in common:

  • Name - set when first referenced and changeable at any time

  • Description - short, unstructured text

  • Tags - a list of tags used for ad hoc grouping of plan elements

  • Categories - a list of same-kind entity types (an agent is categorized by agent types, a role is categorized by other roles etc.)

  • Documentation (attached image, reference document etc.)

Of these common attributes, categories are particularly important. How an agent, role, place, organization etc. is categorized can have wide-ranging consequences on how tasks, which are typically defined in terms of entity types, are assigned, and thus on what information sharing commitments are computed from an IS plan. A small change in the categorization of an organization or a role can significantly alter the IS procedures implied in an IS plan.

Most kinds of entities have their own idiosyncratic attributes. For example, an actual organization can have a parent organization, an actual agent can have security clearances, a communication medium might delegate transmissions to other media etc. The Profiling activities of the ISP Method describe what these attributes are and give their meanings.

Entity profiles are accessed in different ways. An entity's profile can be accessed in Channels:

  • from wherever it is named as part of the definition a plan element,

  • from the plan's index

  • from analytical content, such as the compilations of references (for example, all references to the role "Manager")

As a rule, when the name of an entity (or of its kind) appears in blue, it is a hyperlink that, when clicked, opens a the profile edit form for that entity.

Profile an event

There are only event types in an IS plan.

Table 3.12. Profiling: Profile an event
Prerequisites:

  • The event was named
Steps:

(All steps are optional and they can be taken in any order)

  • Change the name of the event

  • Describe the event

  • Tag the event

  • Categorize the event with other events (avoiding circular categorization)

  • Attach picture or document

  • Identify the location, type or actual, where the event would occur
Outcomes:

  • Depending on its categorization, the event may or may not be compatible as a category for certain other events.

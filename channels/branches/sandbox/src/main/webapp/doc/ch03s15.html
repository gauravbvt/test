Plan definition, management and analytics

A planner selects the item “About Plan” in the top “Show” menu to access the Plan panel where the planner can edit the name, description, planned phases, locale and attached documentation of the Plan.

The Plan panel also gives access to:

  • A list of all events referenced in the Plan, with the ability to declare certain events as "incidents" (i.e. "outside" events not expected to be caused by planned tasks)

  • The list of all secrecy classifications (also referenced in the Plan), with the ability to add and edit secrecy classifications,

  • The list of all organizations in scope, where a planner can indicate if an organization is expected to be assigned planned tasks,

  • A visualization and analysis of how the segments inter-relate

  • A searchable who's who of all participating agents

  • A searchable list of all issues anywhere in the Plan

  • A searchable list of all document attachments

  • An index of all named Plan elements (tasks, flows, entities etc.)

  • An overall evalutation of the Plan

  • A list of all registered users and their identification with agents in the Plan (particiaptions)

  • A mechanism for releasing the current development version of the Plan into production

The default Plan view is the Details view. Here, planners can change the name of a Plan, add phases and remove unreferenced phases. Planners can also add or view issues about the Plan as a whole.

In theAll events view, a planner can add events to the scope of the Plan. A planner can mark an event as an incident if it is caused “outside” of the Plan and not (always) as an outcome of a planned task. There is always at least one event defined in a Plan. Un-marking as incident an event that is not defined and is not referenced anywhere in the Plan removes it from the Plan

In theSecrecy classification view, a planner can view all defined secrecy classifications used to qualify elements of information (classification), transmission media (security) and agents (security clearances). Classifications can be re-ordered, references to a classification can be listed, unreferenced classifications can be removed and new classifications added.

In theAll organizations view, a planner can see all organizations within the scope of the Plan and add new ones, possibly with expectations that they be assigned tasks. An organization in scope that is expected to have tasks but has none raises a completeness issue.

A network view of all inter-organization sharing commitments is available.

In theAll segments view, planners can look at a diagram of all segments in the Plan. The segments can be grouped by phase or by event. Planners can browse all cross-segment sharing flows as well as all tasks in one segment that start or terminate another segment.

In theWho's who view, planners can search a list of all agents involved in the Plan.

In theAll issues view, planners can sort, filter and browse all issues currently detected or manually added within the Plan. Surveys on individual issues can be created or accessed from this view.

In theBibliography view, planners browse all the documents that are attached anywhere in the Plan.

In theIndex view, planners can find, filter, sort and navigate to all names elements (tasks, flows, entities, segments...).

The Evaluation view gives a summary evaluation of the Plan's fitness by counting the numbers of validity, completeness and robustness issues.

The Participations view lists all registered users and identites the agent, if any, that represerents each user. The planner can make or change this identification. A user that is not identified with a participating agent does not have information sharing procedures.

The Version view is where planners vote to move the development version of the Plan into production. A unanimous vote is required. Moving a development version into production creates a development version from a copy of the new production version. Planners can be sent messages to prompt them to vote.
